Chapter 1

設計哲學 — 為什麼需要分層？

"The center of your application is not the database. Nor is it one or more of the frameworks you may be using. The center of your application is the use cases of your application." — Robert C. Martin, Clean Architecture (2017)

當一個 RESTful API 專案從簡單的 CRUD 逐漸成長，最常遇到的問題是：Controller 變成上帝類別——它同時做驗證、業務邏輯、資料存取、回應組裝。一旦要更換資料庫、增加快取、或修改業務規則，你必須翻動整個 Controller，牽一髮而動全身。

本架構的核心信念是：

🎯 關注點分離 (Separation of Concerns)

每一層只負責一件事。Controller 不碰資料庫，Repository 不做業務判斷，Domain 不知道 HTTP 的存在。

🔄 依賴方向單一且可控

外層可以知道內層，內層絕不知道外層。Domain 是宇宙的中心，它不依賴任何人。

🔌 可替換的基礎設施

透過 Port (Interface) 定義契約，Repository 只是「插進去的一個實作」。今天用 MongoDB，明天換 PostgreSQL，Application 層一行都不用改。

給小白

從你熟悉的 MVC 出發 — 一步一步演化到四層架構

如果你第一次接觸 RESTful API，你學的大概是這樣的三層結構： Controller 接請求 → Service 處理邏輯 → Repository 存資料庫。 這沒有錯，這是一個好的起點。但這份文件要介紹的架構多了一個 Domain Layer， 而且 Repository 的依賴方向也「反過來了」。

這章專門解釋：為什麼要這樣演化？每一步改變是為了解決什麼問題？

第一階段：你一開始學的樣子

Controller（接 HTTP）
↓ 呼叫
Service（流程 + 業務規則都在這）
↓ 呼叫
Repository（存資料庫）
@Service
public class UseCouponService {
    public void use(String couponId, String userId) {
        // 1. 去資料庫拿資料
        Coupon c = repo.findById(couponId);

        // 2. 業務判斷 (寫在 Service 裡)
        if (c.isUsed())
            throw new RuntimeException("已使用");

        // 3. 改狀態 (也寫在 Service 裡)
        c.setUsed(true);
        c.setUsedBy(userId);

        // 4. 存回資料庫
        repo.save(c);
    }
}

專案很小的時候，這樣寫完全沒問題。但請想像以下情境：

  • 老闆說：「同一張券，同一個人不能領兩次，而且過期的也不能用。」
  • 你把這些 if 全寫進 Service，Service 越來越長。
  • 下個月有另一個功能也需要「驗證優惠券狀態」，你就又在別的 Service 重寫一次一樣的 if。
  • 產品規則改了，你要找遍所有 Service 去更新那些 if。

🚨 Service 膨脹的症狀

  • 一個 Service 超過 100 行，而且全是 if / else
  • 兩個不同的 Service 有幾乎一樣的業務判斷程式碼（複製貼上）
  • 要測試「優惠券不能重複使用」，卻必須 Mock 資料庫才能跑測試

第二階段：把「業務規則」從 Service 裡拉出來 → Domain 層誕生

關鍵洞察：Service 是「流程協調者」，不是「規則守護者」。

🏗️ 現實比喻：工地包工頭 vs 法規手冊

想像蓋房子。
Service = 工地包工頭：他負責安排「先打地基、再立牆、再蓋屋頂」的流程。
Domain = 建築法規手冊：「牆壁厚度不能低於 15cm」、「逃生出口不能被遮擋」。

包工頭不需要把法規背在腦子裡——他說「按法規蓋」，法規手冊自己判斷對不對。 同理，Service 說 coupon.use(userId)，Coupon 物件自己判斷能不能用。

Controller
↓
Service（只管流程）
↓   ↓
Domain（業務規則住這）
↓
Repository
// Domain 物件：業務規則的家
public class Coupon {
    private boolean used;
    private String ownerId;

    public void use(String userId) {
        // 業務規則 IN 這裡，不在 Service
        if (this.used)
            throw new CouponAlreadyUsedException();
        if (!this.ownerId.equals(userId))
            throw new NotOwnerException();
        this.used = true;
        this.usedBy = userId;
    }
}

// Service：只管流程，不管規則細節
public class UseCouponService {
    public void use(UserId uid, CouponId cid) {
        Coupon c = repo.find(cid);  // 拿
        c.use(uid.getId());         // 規則判斷交給 Domain
        repo.save(c);               // 存
    }
}

現在，「優惠券不能重複使用」這個規則只存在一個地方：Coupon.use()。 不管哪個 Service 要用優惠券，都呼叫這個方法，規則自然被執行。 要測試這個規則，只需要 new Coupon()，完全不需要資料庫。

第三階段：Repository 的「反向依賴」到底在反什麼？

這是最讓小白困惑的部分。讓我們先看「普通寫法」出了什麼問題： 

// ❌ 普通寫法：Service 直接 new 出 Repository
public class UseCouponService {
    private UseCouponMongoRepo repo = new UseCouponMongoRepo();
    //                         ↑ Service 直接綁死在 MongoDB 的實作上
}

這樣的問題是：Service 跟 MongoDB 綁死了。如果你要：

  • 換成 PostgreSQL → Service 要改
  • 寫單元測試用假資料 → Service 要改
  • 加 Redis 快取 → Service 要改
📋 現實比喻：甲方的需求說明書

普通做法（沒有反向依賴）：你是業主（Service），你直接打電話給「王師傅（MongoRepo）」， 你的工作完全依賴王師傅的行程和作法。王師傅退休了，你一切都要重來。

有反向依賴的做法：你是業主（Application），你先寫一份「需求說明書（Port Interface）」： 「我需要一個能找優惠券、能儲存優惠券的服務。」然後貼公告，讓任何符合這份需求的承包商來競標。 王師傅（MongoRepo）、李師傅（MySQLRepo）都可以來，都要照你的需求說明書來做。

「反向」的是：以前是師傅決定提供什麼、你去配合；現在是你決定你要什麼、師傅來配合你。 

// ✅ 反向依賴寫法

// Step 1：Application 層定義「需求說明書」（Port Interface）
public interface UseCoupon {          // ← 這份合約寫在 Application 層
    Coupon find(CouponId id);
    void save(Coupon c);
}

// Step 2：Service 只依賴「需求說明書」，不認識任何具體 Repo
public class UseCouponService {
    private final UseCoupon repo;  // ← 只知道介面，不知道背後是 Mongo 還是 MySQL
}

// Step 3：Repository 層「按照需求說明書」實作
public class UseCouponMongoRepo implements UseCoupon {  // ← Repo 配合 Application 的合約
    public Coupon find(CouponId id) { /* MongoDB 查詢 */ }
    public void save(Coupon c)       { /* MongoDB 寫入 */ }
}

💡 「反向」是指：誰定義介面的歸屬權反了

  • 沒反向前：Repository 寫好自己的 class，Service 去用它 → 「Service 依賴 Repository」
  • 反向後：Application 寫好 Interface，Repository 去實作它 → 「Repository 依賴 Application 的合約」
  • 依賴箭頭從「Service → MongoRepo」變成「MongoRepo → (Interface in Application)」
  • Repository 的程式碼裡有 implements UseCoupon，所以是 Repository 依賴 Application 層的介面，不是反過來

三個階段的完整對比

問題 階段一（純 MVC） 階段二（加 Domain） 階段三（加 Port/DIP）
業務規則放哪？ Service 裡的 if/else Domain 物件的方法 Domain 物件的方法 ✅
換資料庫要改哪？ Service（因為直接 new Repo） Service（仍然直接 new Repo） 只改 Repository 實作 ✅
測試業務規則需要 DB？ 需要（規則在 Service 裡） 不需要（規則在 Domain 裡）✅ 不需要 ✅
測試 Service 需要 DB？ 需要（直接 new MongoRepo） 需要（直接 new MongoRepo） 不需要（注入 Mock）✅

✅ 現在你可以帶著這個理解往下讀

  • Domain = 業務規則的家，不依賴任何外部技術
  • Application (Service) = 流程指揮官：「先拿、讓 Domain 判斷、再存」
  • Port (Interface) = Application 層貼出的「需求說明書」
  • Repository = 按需求說明書實作的承包商，只管存取技術細節
給小白（進階）

Domain 是什麼在流動？ — 資料、狀態、與規則

上一章解釋了 Domain 的存在意義。但很多人看完還是會卡住： 「流程寫在 Application，資料庫寫在 Repository，那 Domain 到底在哪裡活動？ 它到底是個什麼？」

這章用一個生活比喻，再加一個完整的「領券」範例，讓你徹底搞清楚 Domain 物件在系統裡怎麼流動、狀態怎麼變、規則怎麼帶著走。

生活比喻：外送平台點餐

🛵 用外送訂單來理解三層分工

你用 App 點了一份牛肉麵。這個過程有三個角色：

  • 平台客服（Application Layer）：接受你的點餐、安排送餐、確認送達、處理退款——她負責跑整個流程的 SOP。
  • 訂單本身（Domain Object）：從你點單的那一刻起，訂單就存在了。它帶著你的資料、帶著金額、帶著「未送達→送達→已評價」的狀態。它也帶著規則：「已評價的訂單不能再評價」、「已取消的訂單不能送餐」。
  • 訂單資料庫（Repository）：只負責把訂單存進去、取出來。它不管你的訂單能不能取消，那不是它的事。

注意：訂單（Domain 物件）在客服（Application）和資料庫（Repository）之間流動。 客服呼叫資料庫取出訂單，操作訂單的狀態，再請資料庫存回去。訂單本身帶著規則，不管在哪裡被操作，規則都跟著。

「你其實一直都在建立 Domain 物件」

當你第一次寫優惠券時，你可能這樣寫： 

// 小白第一次寫法（你"已經"在描述 Domain，只是沒意識到）
Coupon coupon = new Coupon();
coupon.setCouponId(UUID.randomUUID().toString());
coupon.setUserId(userId);
coupon.setUsed(false);
coupon.setExpiredAt(LocalDate.now().plusDays(30));

// 然後業務判斷寫在 Service 裡：
if (coupon.isUsed()) throw new RuntimeException("已使用");
coupon.setUsed(true);
coupon.setUsedBy(userId);

你已經在描述 Domain！你知道優惠券有 ID、有使用者、有狀態、有有效期—— 這就是 Domain 物件的資料。 問題只有一個：你把規則（if 判斷）和狀態變更（set 操作）放在 Service， 而不是放進 Coupon 物件本身。

Domain layer 要做的事，只是把規則從 Service 移到物件裡： 

// Domain 版：規則住進物件本身
public class Coupon {
    private String couponId;
    private String ownerId;
    private boolean used;
    private LocalDate expiredAt;

    // ① 建立券的工廠方法（帶著初始規則）
    public static Coupon issue(String ownerId) {
        Coupon c = new Coupon();
        c.couponId  = UUID.randomUUID().toString();
        c.ownerId   = ownerId;
        c.used      = false;
        c.expiredAt = LocalDate.now().plusDays(30);
        return c;
    }

    // ② 使用券的業務行為（帶著使用規則）
    public void use(String userId) {
        if (this.used)
            throw new CouponAlreadyUsedException();
        if (LocalDate.now().isAfter(this.expiredAt))
            throw new CouponExpiredException();
        if (!this.ownerId.equals(userId))
            throw new NotOwnerException();
        this.used   = true;
        this.usedBy = userId;
    }
}
// Coupon 就像演唱會票根：票根本身就印著「限本人使用、一次性入場」的規則。
// 不管交給哪個驗票員（Service），規則都在票上，不會搞丟。

完整「領券」流程 — Domain 物件的狀態旅程

現在來看一個完整的「領券（Issue Coupon）」流程， 追蹤 Domain 物件是如何在 Application 和 Repository 之間流動，以及狀態怎麼一步步改變：

步 誰在動作 做了什麼 Domain 物件的狀態
1 Application 呼叫 repo.findUser(userId) User{ id, issuedCount: 2 }
從 DB 取出 User 物件
2 Domain 呼叫 user.canIssueCoupon() User 自己判斷：每人上限 3 張，目前 2 張 → OK
業務規則在 User 物件裡，不在 Service 裡
3 Domain 呼叫 Coupon.issue(userId) Coupon{ id: 新ID, used: false, expiredAt: +30天 }
Domain 物件誕生，帶著初始狀態
4 Domain 呼叫 user.recordIssuance() User{ id, issuedCount: 3 }
User 物件的計數器 +1（Domain 狀態更新）
5 Application 呼叫 repo.save(coupon) Coupon 物件交給 Repository，Repository 把它轉換成 DB 格式存入
Repository 只管「怎麼存」，不管「能不能存」
6 Application 呼叫 repo.saveUser(user) 更新後的 User 物件也存回 DB
流程結束，所有狀態持久化

整個流程的程式碼就是這樣：

@Service
public class IssueCouponService {

    public void issue(UserId userId) {
        // 1. Repo 把 User 從 DB 取出 → 交給 Application
        User user = userRepo.find(userId);

        // 2. Application 把 User 交給 Domain 判斷
        user.canIssueCoupon();      // User 自己知道規則，不行就拋例外

        // 3. Domain 建立新的 Coupon 物件（帶著初始狀態）
        Coupon coupon = Coupon.issue(userId.getId());

        // 4. Domain 更新 User 狀態
        user.recordIssuance();

        // 5 & 6. Application 把 Domain 物件交給 Repo 存回 DB
        couponRepo.save(coupon);
        userRepo.save(user);
    }
}

// Service 做的事只有：拿 → 交給 Domain 操作 → 存
// 所有業務判斷和狀態變更，都在 Domain 物件的方法裡完成
// Service 自己沒有一個 if/else

生活中早就存在的「Domain 物件」

Domain layer 之所以感覺「多出來」又「難理解」， 是因為把規則放進物件本身這件事在現實世界中其實天然存在， 只是我們寫程式時沒有把這個直覺帶進來。

生活中的 Domain 物件 物件自帶的規則（Domain Rules） 誰只管「存取」（Repository 概念）
🎫 演唱會票根 「每張票限入場一次」
「入場後不退票」
「持票者本人才能入場」		 票務系統資料庫（存票、查票）
🏦 銀行帳戶 「餘額不能為負」
「每日轉帳上限 100 萬」
「凍結帳戶不能轉出」		 銀行核心系統（存餘額、查紀錄）
🅿️ 停車場票根 「入場時間印在票上決定計費」
「同一張票不能入場兩次」		 停車場管理系統（存入場紀錄）
📚 圖書館借書證 「同本書同時只有一人可借」
「逾期未還要罰款」
「最多借 5 本」		 圖書館系統（存借還紀錄）
🎟️ 優惠券 「每人限領一次」
「使用後不可再用」
「有效期 30 天」		 優惠券資料庫（存券、查券）

在現實世界，這些規則印在物件上（票根背面的使用說明） 或寫在合約裡（帳戶開戶條款）。 規則跟著物件走，不管哪個「服務流程」（Application）在處理這個物件，規則都在。 這就是 Domain layer 要模擬的現實。

為什麼「把規則放進物件」這麼反直覺？

因為你學程式是從「一步一步執行的流程」開始學的： 先做 A，再做 B，再做 C。這個思考模式很自然地會把判斷寫在流程裡，而不是物件裡。

但想想你說中文的方式：你會說「這張票已經用過了，不能再用」—— 主詞是「票」，規則是票的屬性。你不會說「如果這張票的 used 欄位是 true，那就不能用」—— 那是把規則從物件身上剝離出來，放到外面用 if 表達。

💡 把「說話方式」帶回程式

  • 中文說：「這張票已經用過了」→ 程式寫：ticket.use()（票自己知道能不能用）
  • 中文說：「帳戶餘額不足」→ 程式寫：account.withdraw(amount)（帳戶自己判斷能不能扣）
  • 中文說：「優惠券還在有效期」→ 程式寫：coupon.isValid()（券自己知道自己的狀態）

當你的程式碼可以像說中文一樣理解，就代表 Domain 設計是正確的。

一眼看出邊界問題的快速方法

你在哪裡看到這段程式 邊界正確嗎？ 應該怎麼做
Service.java 裡有 coupon.isUsed() 的 if 判斷 ❌ 邊界錯誤 這個判斷移進 Coupon.use()
Service.java 裡有 coupon.setUsed(true) ❌ 邊界錯誤 狀態變更封進 Coupon.use()
Repo.java 裡有 if (alreadyIssued) throw ❌ 邊界錯誤 業務判斷移到 Domain 或 Application
Domain Coupon.java 裡有 mongoTemplate.find() ❌ 邊界錯誤 Domain 不能碰 DB，查詢移到 Repository
Service.java 裡只有：拿 → 呼叫 Domain 方法 → 存 ✅ 正確 Service 是純流程協調，沒有業務 if
Domain Coupon.java 裡沒有任何 import 外部框架 ✅ 正確 Domain 是純 Java，不依賴任何技術
Chapter 2

核心原則

依賴反轉原則 (Dependency Inversion Principle)

高層模組不應該依賴低層模組，兩者都應該依賴抽象。在本架構中，Application Layer 定義 Port（Interface），Repository Layer 實作它。Application 永遠只面對介面程式設計，從不直接 new 出一個 Repository。

🎯 「高層」vs「低層」是什麼意思？

「高/低」描述的是抽象程度，不是圖中的大小或位置。對應到後面的同心圓：

  • 高層模組（內圈）：Domain、Application — 純業務邏輯，不在乎資料怎麼存、用哪個框架
  • 低層模組（外圈）：Repository、Web — 技術實作細節，知道 MySQL / HTTP 的存在

程式小白直覺上可能覺得「高 = 外圈（大）」，但在架構術語中恰恰相反：越抽象、越靠近業務核心，就叫「高層」。

單一職責原則 (Single Responsibility Principle)

每個 Controller 只處理一個 Use Case。每個 Service 協調一個業務流程。每個 Repository 負責一種資料存取。當需求變更時，你可以精確定位到該改哪一個類別。

介面隔離原則 (Interface Segregation Principle)

Port 按照使用者（Service）的需求來切割，而不是按照實作方的方便。一個 Service 需要「寫入 bitmap」和「讀取 bitmap」兩件事，就定義兩個方法的 Port，不會塞入 20 個不相關的方法。

Chapter 4

Web Layer（表現層）

Web Layer 是系統的「門面」，它唯一的工作是：接收 HTTP、驗證格式、轉換物件、呼叫 Service、組裝回應。它不包含任何業務邏輯判斷。

Controller — 一個 Use Case 一個 Controller

不採用一個大 Controller 包 N 個方法的做法。每個 Controller 只有一個公開方法，對應一個明確的業務用例。這讓每個 Controller 極度精簡且職責單一。 

@RestController
public class UseCouponController {

    private final UseCouponService service;

    public UseCouponController(UseCouponService service) {
        this.service = service;
    }

    @PostMapping("/coupons/use")
    public UseCouponResponse use(@RequestBody @Valid UseCouponRequest req) {
        // 1. 從 Request DTO 取值，轉為 Application 層能理解的物件
        var result = service.use(
            new UserId(req.getUserId()),
            new CouponId(req.getCouponId())
        );
        // 2. 組裝 Response DTO 回傳
        return new UseCouponResponse(result);
    }
}

port/ 子目錄 — Request & Response DTO

所有進出 Web 層的資料物件統一放在 web/port/ 下。Request 物件承載驗證邏輯（@NotBlank、@NotEmpty 等），Response 物件定義 API 回傳格式。這些 DTO 只存在於 Web 層，不會滲透到其他層。 

public class UseCouponRequest {
    @NotBlank
    private String couponId;
    @NotBlank
    private String userId;
    // getters & setters
}

💡 設計要點

  • Controller 不做 if/else 業務判斷
  • Controller 不直接操作 Repository
  • Request/Response DTO 只在 Web 層使用，不傳入 Service
  • 轉換過程在 Controller 中完成：Request → Domain Object → Service 呼叫
Chapter 5

Application Layer（應用層）

Application Layer 是整個系統的「指揮官」。它不直接做事，而是協調：告訴 Domain 物件做業務邏輯，告訴 Repository（透過 Port）去存取資料。它像一個劇本導演——不演戲，但決定誰在什麼時候上場。

Service — 業務流程協調者

@Service
public class UseCouponService {

    private final UseCoupon repo;  // Port (Interface)，不是具體類別

    public UseCouponService(UseCoupon repo) {
        this.repo = repo;
    }

    public Object use(UserId userId, CouponId couponId) {
        // 1. 透過 Port 取得 Domain 物件
        Coupon c = repo.find(couponId);
        // 2. 呼叫 Domain 物件的業務方法
        c.use(userId.getId());
        // 3. 透過 Port 持久化
        repo.save(c);
        return null;
    }
}

port/ 子目錄 — 抽象介面（依賴反轉的關鍵）

Port 是 Application 層對外部世界的「契約宣告」。它說：「我需要一個能 find 和 save Coupon 的東西」，至於是 MongoDB、MySQL、還是記憶體實作，它不在乎。 

public interface UseCoupon {
    Coupon find(CouponId couponId);
    void save(Coupon c);
}

dto/ 子目錄 — 層間傳遞物件

當原始型別（String）不足以表達語意時，用 DTO 或 Value Object 包裝。UserId 和 CouponId 就是這樣的角色——它們讓方法簽名自我說明，避免參數順序搞混。 

public class CouponId {
    private String id;

    public CouponId(String id) {
        this.id = id;
    }
    // getter
}
Chapter 6

Domain Layer（領域層）

"Domain 才是真正的業務邏輯區域，它包含了業務邏輯的核心概念和規則。它不做跟 infra 或 service layer 有關的事情，就是一個類別做好自己的事。" — 本專案 Coupon.java 中的原始註解

Domain Layer 是整個架構中最穩定、最純粹的一層。它不依賴 Spring、不依賴資料庫驅動、不依賴任何框架。它是純粹的 Java 類別，只表達「這個業務世界是什麼樣子」以及「業務規則是什麼」。

Entity — 有身份、有行為的業務物件

public class Coupon {
    private String couponId;
    private String userId;
    private boolean used;

    public Coupon(String couponId, String userId) {
        this.couponId = couponId;
    }

    // 業務行為：核銷優惠券
    public void use(String userId) {
        used = true;
        this.userId = userId;
    }
}

注意 use() 方法。核銷的邏輯寫在 Domain 物件自身——這是 Domain-Driven Design 的核心精神：行為跟著資料走，而不是把邏輯散落在 Service 或 Controller 中。

✅ Domain 層的特徵

  • 零框架依賴（沒有 @Service、@Repository、@Autowired）
  • 可以用純 JUnit 測試，不需啟動 Spring Context
  • 類別自己管理自己的狀態一致性
  • 業務規則的「唯一真理來源」
Chapter 7

Repository Layer（基礎設施層）

Repository Layer 是「髒活」的所在地——所有跟具體技術相關的事情都在這裡：SQL、MongoDB Driver API、Redis Client、HTTP Client 呼叫外部服務。它的存在意義是：把技術細節關在籠子裡，不讓它污染業務邏輯。

實作 Port Interface

@Repository
public class UseCouponRepo implements UseCoupon {

    public Coupon find(CouponId couponId) {
        // 具體的 DB 查詢邏輯
        // MongoDB / JPA / JDBC ... 在這裡做
    }

    public void save(Coupon c) {
        // 具體的 DB 寫入邏輯
    }
}

Spring 的依賴注入會自動將這個 @Repository 類別注入到 Service 中宣告的 UseCoupon 介面欄位。Service 永遠不知道背後是 MongoDB 還是 MySQL——這就是依賴反轉的威力。

邊界判斷

這段程式碼屬於哪一層？

「我知道每層叫什麼，但我還是不知道這行程式碼該寫在哪裡。」 — 每一位初學架構的工程師

看完前幾章，你知道了 Domain、Application、Repository 的定義。但實際寫程式時，最常出現的問題是： 「這個 if 判斷應該寫在 Coupon.java 還是 UseCouponRepo.java？」 本章給你一個可以隨時拿出來用的判斷工具。

一句話判斷法

🔑 判斷問題：「把整個資料庫換掉，這行程式碼需要跟著改嗎？」

  • 不需要改（跟資料庫無關）→ 業務邏輯 → 放 Domain 或 Application
  • 需要改（跟 SQL / MongoDB API / Redis API 有關）→ 技術細節 → 放 Repository

什麼叫「業務判斷」？

想像你是這家電商公司的老闆，你對工程師說的規則，就是業務規則：

老闆說的規則 層次 換資料庫會變嗎？
「一張優惠券只能被使用一次」 Domain ❌ 不會，這是商業規則
「過期的優惠券不能使用」 Domain ❌ 不會，跟資料庫完全無關
「只有持有人才能使用自己的券」 Domain ❌ 不會，這是身份驗證邏輯
「先查 Redis 快取，沒有再去 MySQL」 Repository ✅ 會，完全是技術選型問題
「用 couponId 當 key 查 MongoDB」 Repository ✅ 會，MongoDB API 是技術細節
「先查券、再核銷、再存回去」的步驟順序 Application ❌ 不會，但這是流程協調，不是核心規則
「HTTP 請求的 couponId 欄位不能空白」 Web ❌ 不會，但這是輸入格式驗證，不是業務規則

Repository 的「最底層」是什麼？

Repository 是整個系統的倉庫管理員。他只做兩件事：

  • 拿資料：去資料庫/外部服務把資料取出來，轉換成 Domain 物件交給你
  • 存資料：把 Domain 物件轉換成資料庫格式，寫進去

他不決定「能不能」做這件事。你叫他拿 coupon-001，他就去拿，拿不到才告訴你找不到。你叫他存，他就存。業務上「這張券可不可以被使用」不是他的問題——那是 Domain 的事。

錯誤 vs 正確：以「使用優惠券」為例

下面展示同一個功能的兩種寫法——同樣能跑，但架構完全不同：

❌ 錯誤：業務判斷混入 Repository
@Repository
public class UseCouponRepo {

    public void use(String couponId, String userId) {
        Coupon c = db.findById(couponId);

        // ❌ 業務判斷不該在這裡
        if (c.isUsed())
            throw new AlreadyUsedException();

        // ❌ 業務行為不該在這裡
        c.setUsed(true);
        c.setUsedBy(userId);

        db.save(c); // ✓ 只有這行是 Repo 的本分
    }
}

// 問題：換資料庫時，業務規則被迫跟著動
// 問題：Domain 物件 Coupon 變成只是資料容器
// 問題：無法用純 JUnit 測試業務規則
✅ 正確：每層各司其職
// --- Domain 層：業務規則的家 ---
public class Coupon {
    public void use(String userId) {
        // ✅ 業務判斷在這裡
        if (this.used)
            throw new CouponAlreadyUsedException();
        if (!this.ownerId.equals(userId))
            throw new NotOwnerException();
        // ✅ 業務行為在這裡
        this.used = true;
        this.usedBy = userId;
    }
}

// --- Repository 層：只管存取 ---
public class UseCouponRepo implements UseCoupon {
    public Coupon find(CouponId id) {
        return db.findById(id.getId()); // 只管拿
    }
    public void save(Coupon c) {
        db.save(c); // 只管存
    }
}

// --- Application 層：指揮流程 ---
public class UseCouponService {
    public void use(UserId uid, CouponId cid) {
        Coupon c = repo.find(cid);  // 叫 Repo 拿
        c.use(uid.getId());         // 叫 Domain 執行
        repo.save(c);               // 叫 Repo 存
    }
}

✅ 正確寫法的三大好處

  • 換掉 MongoDB 改用 MySQL？只改 Repository，Domain 的業務規則一行都不動
  • 要測試「券不能被重複使用」？直接 new Coupon() 然後呼叫 use()，不需要啟動資料庫
  • 增加新規則「VIP 才能用」？只改 Coupon.use()，其餘層不受影響

常見混淆 Q&A

Q：「找不到優惠券就拋出例外」放哪一層？
Repository 負責查，如果查無結果，拋出 NotFoundException 是 OK 的——因為「找不到」是一個技術事實。 但如果業務說「找不到就當作已使用」，那這個業務決定應該在 Application 層處理，Repository 只負責告知「沒找到」這個事實。
Q：「金額必須大於 0」這個驗證放哪裡？
這是兩種不同性質的驗證，分開放：
• Web 層（@Min(1)）：防止外部傳來非法格式，是輸入保護。
• Domain 層：「折扣金額不能超過商品原價」這種業務規則，只有在 Domain 才有這個上下文（因為需要知道「原價」這個業務概念）。
兩者可以並存，目的不同：Web 層防壞資料進來，Domain 層確保業務正確性。
Q：資料庫的 Unique Constraint 算業務邏輯嗎？
不算。資料庫約束是「最後一道防線」的技術保護，不是業務邏輯的表達方式。業務規則「每人只能領一次優惠券」應該在 Domain 或 Application 明確寫出來，不應該讓資料庫拋出 DuplicateKeyException 來代替你表達業務規則——那樣的話，業務規則就藏在資料庫設定裡，看程式碼的人完全看不到。
Q：「先查 Redis，沒有再查 MySQL」這個判斷屬於哪層？
Repository 層。這是技術策略（快取機制），跟業務規則無關。換快取方案或拿掉 Redis，只需要改 Repository；Service 不需要知道背後用了幾個儲存技術。
Q：我怎麼知道某個 if 判斷是「業務」還是「技術」？
問你自己：「這個 if 裡面的條件，非技術人員（PM、老闆）能理解且認可嗎？」
• 「優惠券已使用不能再用」→ PM 說得出來 → 業務規則 → Domain
• 「Redis TTL 過期就重查 MySQL」→ PM 不在乎怎麼查 → 技術細節 → Repository

Port & Adapter 機制詳解

"Allow an application to equally be driven by users, programs, automated test or batch scripts, and to be developed and tested in isolation from its eventual run-time devices and databases." — Alistair Cockburn, Hexagonal Architecture (2005)

本架構借鑑 Cockburn 的 Hexagonal Architecture，使用「Port」和「Adapter」來切斷層與層之間的編譯期耦合：

概念 在本架構中的對應 位置
Port Interface（如 UseCoupon, IssueCouponRedis） application/port/
Adapter 具體實作（如 UseCouponRepo, IssueCouponRedisRepo） repository/
Driving Side Controller（驅動應用程式的外部力量） web/
Driven Side Repository（被應用程式驅動的外部資源） repository/

Port 的定義權在 Application 層（內層），實作責任在 Repository 層（外層）。這確保了依賴箭頭指向內層——外層實作內層定義的契約，而不是內層去配合外層的 API。

Chapter 11

實作範例 — Issue Coupon（多 Port 協調）

真實世界中，一個業務操作往往需要操作多個基礎設施。「發放優惠券」需要同時寫入 MongoDB（持久化）和 Redis（快取加速查詢）。本架構優雅地處理這個場景：定義多個 Port，由 Service 協調。 

@Service
public class IssueCouponService {

    private final IssueCouponMongo mongoPort;   // Port 1: MongoDB
    private final IssueCouponRedis redisPort;   // Port 2: Redis

    public IssueCouponService(IssueCouponMongo mongoPort,
                              IssueCouponRedis redisPort) {
        this.mongoPort = mongoPort;
        this.redisPort = redisPort;
    }

    public int issue(String cpCode, List<String> custNos) {
        int bitmapValue = encodeCpCode(cpCode);

        for (String custNo : custNos) {
            RoaringBitmap bitmap = loadOrCreateBitmap(custNo);
            bitmap.add(bitmapValue);
            byte[] bytes = serializeBitmap(bitmap);

            // 協調兩個 Port 完成寫入
            mongoPort.upsertBitmap(custNo, bytes);  // 持久層
            redisPort.writeBitmap(key, bytes);       // 快取層
        }
        return bitmapValue;
    }
}

Service 的角色在此更為明顯——它是「協調者」，安排 MongoDB 和 Redis 的操作順序。如果未來要加入「發送通知」，只需再定義一個 Port，注入即可，現有程式碼不受影響。

💡 多 Port 的優勢

  • 每個 Port 可以獨立測試（Mock 一個，真的跑另一個）
  • 新增基礎設施只需：定義 Port → 實作 Repo → 注入 Service
  • 不同 Port 可以用不同技術（Mongo + Redis + HTTP + MQ...）

Chapter 13

禁忌與反模式

🚫 絕對不要做的事

  • Controller 直接呼叫 Repository — 繞過 Application 層意味著業務邏輯散落各處
  • Service 直接 new Repository — 失去依賴反轉的好處，無法替換、無法測試
  • Domain 物件 import Spring 類別 — Domain 一旦依賴框架，就不再「純粹」
  • Repository 做業務判斷 — 「如果餘額不足就不扣款」這種邏輯屬於 Domain 或 Service
  • 把 Request DTO 傳進 Service — 這讓 Application 層被 Web 層汙染
  • 一個 Controller 做十件事 — 違反 SRP，應該拆成多個 Controller

每條禁忌的背後都是同一個理由：維持層間邊界的清晰。一旦邊界被突破，技術債務就會像破窗一樣蔓延——「反正那邊已經亂了，這裡也亂一下吧」。
