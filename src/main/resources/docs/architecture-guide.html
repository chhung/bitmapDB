Chapter 1

設計哲學 — 為什麼需要分層？

"The center of your application is not the database. Nor is it one or more of the frameworks you may be using. The center of your application is the use cases of your application." — Robert C. Martin, Clean Architecture (2017)

當一個 RESTful API 專案從簡單的 CRUD 逐漸成長，最常遇到的問題是：Controller 變成上帝類別——它同時做驗證、業務邏輯、資料存取、回應組裝。一旦要更換資料庫、增加快取、或修改業務規則，你必須翻動整個 Controller，牽一髮而動全身。

本架構的核心信念是：

🎯 關注點分離 (Separation of Concerns)

每一層只負責一件事。Controller 不碰資料庫，Repository 不做業務判斷，Domain 不知道 HTTP 的存在。

🔄 依賴方向單一且可控

外層可以知道內層，內層絕不知道外層。Domain 是宇宙的中心，它不依賴任何人。

🔌 可替換的基礎設施

透過 Port (Interface) 定義契約，Repository 只是「插進去的一個實作」。今天用 MongoDB，明天換 PostgreSQL，Application 層一行都不用改。

Chapter 2

核心原則

依賴反轉原則 (Dependency Inversion Principle)

高層模組不應該依賴低層模組，兩者都應該依賴抽象。在本架構中，Application Layer 定義 Port（Interface），Repository Layer 實作它。Application 永遠只面對介面程式設計，從不直接 new 出一個 Repository。

單一職責原則 (Single Responsibility Principle)

每個 Controller 只處理一個 Use Case。每個 Service 協調一個業務流程。每個 Repository 負責一種資料存取。當需求變更時，你可以精確定位到該改哪一個類別。

介面隔離原則 (Interface Segregation Principle)

Port 按照使用者（Service）的需求來切割，而不是按照實作方的方便。一個 Service 需要「寫入 bitmap」和「讀取 bitmap」兩件事，就定義兩個方法的 Port，不會塞入 20 個不相關的方法。

Chapter 4

Web Layer（表現層）

Web Layer 是系統的「門面」，它唯一的工作是：接收 HTTP、驗證格式、轉換物件、呼叫 Service、組裝回應。它不包含任何業務邏輯判斷。

Controller — 一個 Use Case 一個 Controller

不採用一個大 Controller 包 N 個方法的做法。每個 Controller 只有一個公開方法，對應一個明確的業務用例。這讓每個 Controller 極度精簡且職責單一。 

@RestController
public class UseCouponController {

    private final UseCouponService service;

    public UseCouponController(UseCouponService service) {
        this.service = service;
    }

    @PostMapping("/coupons/use")
    public UseCouponResponse use(@RequestBody @Valid UseCouponRequest req) {
        // 1. 從 Request DTO 取值，轉為 Application 層能理解的物件
        var result = service.use(
            new UserId(req.getUserId()),
            new CouponId(req.getCouponId())
        );
        // 2. 組裝 Response DTO 回傳
        return new UseCouponResponse(result);
    }
}

port/ 子目錄 — Request & Response DTO

所有進出 Web 層的資料物件統一放在 web/port/ 下。Request 物件承載驗證邏輯（@NotBlank、@NotEmpty 等），Response 物件定義 API 回傳格式。這些 DTO 只存在於 Web 層，不會滲透到其他層。 

public class UseCouponRequest {
    @NotBlank
    private String couponId;
    @NotBlank
    private String userId;
    // getters & setters
}

💡 設計要點

  • Controller 不做 if/else 業務判斷
  • Controller 不直接操作 Repository
  • Request/Response DTO 只在 Web 層使用，不傳入 Service
  • 轉換過程在 Controller 中完成：Request → Domain Object → Service 呼叫
Chapter 5

Application Layer（應用層）

Application Layer 是整個系統的「指揮官」。它不直接做事，而是協調：告訴 Domain 物件做業務邏輯，告訴 Repository（透過 Port）去存取資料。它像一個劇本導演——不演戲，但決定誰在什麼時候上場。

Service — 業務流程協調者

@Service
public class UseCouponService {

    private final UseCoupon repo;  // Port (Interface)，不是具體類別

    public UseCouponService(UseCoupon repo) {
        this.repo = repo;
    }

    public Object use(UserId userId, CouponId couponId) {
        // 1. 透過 Port 取得 Domain 物件
        Coupon c = repo.find(couponId);
        // 2. 呼叫 Domain 物件的業務方法
        c.use(userId.getId());
        // 3. 透過 Port 持久化
        repo.save(c);
        return null;
    }
}

port/ 子目錄 — 抽象介面（依賴反轉的關鍵）

Port 是 Application 層對外部世界的「契約宣告」。它說：「我需要一個能 find 和 save Coupon 的東西」，至於是 MongoDB、MySQL、還是記憶體實作，它不在乎。 

public interface UseCoupon {
    Coupon find(CouponId couponId);
    void save(Coupon c);
}

dto/ 子目錄 — 層間傳遞物件

當原始型別（String）不足以表達語意時，用 DTO 或 Value Object 包裝。UserId 和 CouponId 就是這樣的角色——它們讓方法簽名自我說明，避免參數順序搞混。 

public class CouponId {
    private String id;

    public CouponId(String id) {
        this.id = id;
    }
    // getter
}
Chapter 6

Domain Layer（領域層）

"Domain 才是真正的業務邏輯區域，它包含了業務邏輯的核心概念和規則。它不做跟 infra 或 service layer 有關的事情，就是一個類別做好自己的事。" — 本專案 Coupon.java 中的原始註解

Domain Layer 是整個架構中最穩定、最純粹的一層。它不依賴 Spring、不依賴資料庫驅動、不依賴任何框架。它是純粹的 Java 類別，只表達「這個業務世界是什麼樣子」以及「業務規則是什麼」。

Entity — 有身份、有行為的業務物件

public class Coupon {
    private String couponId;
    private String userId;
    private boolean used;

    public Coupon(String couponId, String userId) {
        this.couponId = couponId;
    }

    // 業務行為：核銷優惠券
    public void use(String userId) {
        used = true;
        this.userId = userId;
    }
}

注意 use() 方法。核銷的邏輯寫在 Domain 物件自身——這是 Domain-Driven Design 的核心精神：行為跟著資料走，而不是把邏輯散落在 Service 或 Controller 中。

✅ Domain 層的特徵

  • 零框架依賴（沒有 @Service、@Repository、@Autowired）
  • 可以用純 JUnit 測試，不需啟動 Spring Context
  • 類別自己管理自己的狀態一致性
  • 業務規則的「唯一真理來源」
Chapter 7

Repository Layer（基礎設施層）

Repository Layer 是「髒活」的所在地——所有跟具體技術相關的事情都在這裡：SQL、MongoDB Driver API、Redis Client、HTTP Client 呼叫外部服務。它的存在意義是：把技術細節關在籠子裡，不讓它污染業務邏輯。

實作 Port Interface

@Repository
public class UseCouponRepo implements UseCoupon {

    public Coupon find(CouponId couponId) {
        // 具體的 DB 查詢邏輯
        // MongoDB / JPA / JDBC ... 在這裡做
    }

    public void save(Coupon c) {
        // 具體的 DB 寫入邏輯
    }
}

Spring 的依賴注入會自動將這個 @Repository 類別注入到 Service 中宣告的 UseCoupon 介面欄位。Service 永遠不知道背後是 MongoDB 還是 MySQL——這就是依賴反轉的威力。

Chapter 8

Port & Adapter 機制詳解

"Allow an application to equally be driven by users, programs, automated test or batch scripts, and to be developed and tested in isolation from its eventual run-time devices and databases." — Alistair Cockburn, Hexagonal Architecture (2005)

本架構借鑑 Cockburn 的 Hexagonal Architecture，使用「Port」和「Adapter」來切斷層與層之間的編譯期耦合：

概念 在本架構中的對應 位置
Port Interface（如 UseCoupon, IssueCouponRedis） application/port/
Adapter 具體實作（如 UseCouponRepo, IssueCouponRedisRepo） repository/
Driving Side Controller（驅動應用程式的外部力量） web/
Driven Side Repository（被應用程式驅動的外部資源） repository/

Port 的定義權在 Application 層（內層），實作責任在 Repository 層（外層）。這確保了依賴箭頭指向內層——外層實作內層定義的契約，而不是內層去配合外層的 API。

實作範例 — Use Coupon（單一 Port）

這是最基礎的模式：一個 Controller → 一個 Service → 一個 Port → 一個 Repository。

檔案清單

層 檔案 角色
Web CouponController.java POST /coupons/use 端點
Web web/port/UseCouponRequest.java 輸入驗證
Web web/port/UseCouponResponse.java 回應格式
Application UseCouponService.java 協調 find → use → save
Application application/port/UseCoupon.java Port Interface

Application application/dto/CouponId.java Value Object

這個範例展示了最純粹的 Port-Adapter 模式：Service 注入 UseCoupon 介面，透過它取得 Domain 物件，執行業務邏輯，再透過它持久化。整個流程中 Service 完全不知道背後是什麼資料庫。

Chapter 11

實作範例 — Issue Coupon（多 Port 協調）

真實世界中，一個業務操作往往需要操作多個基礎設施。「發放優惠券」需要同時寫入 MongoDB（持久化）和 Redis（快取加速查詢）。本架構優雅地處理這個場景：定義多個 Port，由 Service 協調。 

@Service
public class IssueCouponService {

    private final IssueCouponMongo mongoPort;   // Port 1: MongoDB
    private final IssueCouponRedis redisPort;   // Port 2: Redis

    public IssueCouponService(IssueCouponMongo mongoPort,
                              IssueCouponRedis redisPort) {
        this.mongoPort = mongoPort;
        this.redisPort = redisPort;
    }

    public int issue(String cpCode, List<String> custNos) {
        int bitmapValue = encodeCpCode(cpCode);

        for (String custNo : custNos) {
            RoaringBitmap bitmap = loadOrCreateBitmap(custNo);
            bitmap.add(bitmapValue);
            byte[] bytes = serializeBitmap(bitmap);

            // 協調兩個 Port 完成寫入
            mongoPort.upsertBitmap(custNo, bytes);  // 持久層
            redisPort.writeBitmap(key, bytes);       // 快取層
        }
        return bitmapValue;
    }
}

Service 的角色在此更為明顯——它是「協調者」，安排 MongoDB 和 Redis 的操作順序。如果未來要加入「發送通知」，只需再定義一個 Port，注入即可，現有程式碼不受影響。

💡 多 Port 的優勢

  • 每個 Port 可以獨立測試（Mock 一個，真的跑另一個）
  • 新增基礎設施只需：定義 Port → 實作 Repo → 注入 Service
  • 不同 Port 可以用不同技術（Mongo + Redis + HTTP + MQ...）

📐 放置規則

  • Port Interface 永遠放在 application/port/ — 因為它屬於「應用層的需求宣告」
  • Web DTO 永遠放在 web/port/ — 明確屬於表現層
  • Application DTO 放在 application/dto/ — 層間通訊用
  • Domain 物件 放在 domain/ — 乾淨、無依賴
  • 所有實作 放在 repository/ — 不管是 DB、Cache 還是外部 API
Chapter 13

禁忌與反模式

🚫 絕對不要做的事

  • Controller 直接呼叫 Repository — 繞過 Application 層意味著業務邏輯散落各處
  • Service 直接 new Repository — 失去依賴反轉的好處，無法替換、無法測試
  • Domain 物件 import Spring 類別 — Domain 一旦依賴框架，就不再「純粹」
  • Repository 做業務判斷 — 「如果餘額不足就不扣款」這種邏輯屬於 Domain 或 Service
  • 把 Request DTO 傳進 Service — 這讓 Application 層被 Web 層汙染
  • 一個 Controller 做十件事 — 違反 SRP，應該拆成多個 Controller

每條禁忌的背後都是同一個理由：維持層間邊界的清晰。一旦邊界被突破，技術債務就會像破窗一樣蔓延——「反正那邊已經亂了，這裡也亂一下吧」。
